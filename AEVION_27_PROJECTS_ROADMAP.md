# AEVION — 27 проектов: дорожная карта до рабочих версий

**База:** список в `aevion-globus-backend/src/data/projects.ts` (ровно **27** узлов).  
**«Рабочая версия» (Working v1):** для пользователя есть стабильный сценарий (UI или API), данные не теряются при перезапуске там, где нужна персистентность, есть базовая документация запуска.

**Оценка сроков** — при **1 основном разработчике + ИИ-ассистент**, фокус на релизных инкрементах без полной «идеальной» фичи. Реальность: параллельные обязанности и непредвиденные интеграции сдвигают на **+25–40%**.

**Процесс (быстрее всего к рабочим версиям):** см. **`AEVION_COORDINATION.md`** — раздел *«Рекомендуемая стратегия»* (2 чата: SYSTEM + SPRINT, один вертикальный срез за спринт, `npm run verify`).

---

## Ускорение: максимум смысла за единицу времени

При **неизменных приоритетах** сроки сокращаются не «параллельным написанием 27 продуктов», а за счёт **платформы и вертикальных срезов**.

1. **Единый продуктовый контур (уже в UI)**  
   Любой модуль `/${id}` ведёт через один сценарий: **Auth → QRight (карточка + локация) → QSign (префилл JSON) → IP Bureau (deep link с `country`/`city`)**.  
   Пока нет отдельной страницы модуля, пользователь всё равно получает **осмысленный** шаг: идентичность, реестр, целостность, бюро.

2. **Вертикальный срез вместо «доделать идеально слой X»**  
   На каждый спринт: **один модуль = один сценарий E2E** (например: «создать QRight → подписать → проверить»), а не три недели только визуала Globus.

3. **Повторное использование обязательно**  
   Любой новый модуль подключает **тот же** `NEXT_PUBLIC_API_BASE_URL`, Auth token, QSign payload shape (`aevion.module.stub.v1` или преемник). Новый код — только **доменная логика**, не новый auth.

4. **Жёсткий scope на MVP**  
   В Working v1 не входят: реки на глобусе, идеальный UX, все edge cases. Входит: **стабильный happy path + данные в БД** там, где обещали персистентность.

5. **Параллель только там, где нет зависимости**  
   Например, контент/копирайт Kids AI и тесты QTrade можно вести параллельно; **QPayNet** — после модели данных QTrade или общего «ledger»-контракта.

6. **ИИ в конвейере**  
   Шаблоны роутов, OpenAPI-заготовки, тесты и миграции — генерировать пакетами под **один** выбранный модуль за итерацию, а не размазывать по всем 27.

7. **Один gate на сборку**  
   Из корня репозитория: **`npm run verify`** — последовательно собирает backend (`tsc`) и frontend (`next build`). Меньше «забыл собрать вторую половину» перед демо или merge.

8. **Параллель в CI**  
   Workflow **`.github/workflows/ci.yml`**: backend и frontend билдятся **одновременно** на GitHub — быстрее обратная связь, чем один длинный job.

9. **Один терминал на dev**  
   **`npm run dev`** в корне (через `concurrently`) — backend + frontend без ручного переключения папок. Подробности — **`README.md`** в корне `aevion-core`.

*Ожидаемый эффект:* на **20–35%** быстрее выход «осмысленного» MVP по модулям за счёт отказа от уникального каркаса в каждом продукте и фокуса на E2E-срезах (при тех же приоритетах). Дополнительно **инструменты (п.7–9)** экономят часы на рутине и регрессах при каждом изменении.

## Если цель — «все 27 хоть как-то кликабельны»

- **Только витрина** (страница `/[id]` + описание + ссылка «в разработке»): можно **за 1–2 недели** массово.  
- **Минимально осмысленный сценарий в каждом** (форма + API + БД где нужно): порядка **18–30 месяцев** последовательно или **12–18** при 2 FTE и жёсткой приоритизации.
